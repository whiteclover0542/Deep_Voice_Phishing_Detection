## 탐지 결과

### 테스트 구성
- **진짜 파일**: 실제 전화 통화 녹음본 사용
- **가짜 파일**: edge-tts(Microsoft)로 보이스피싱 시나리오 문장을 직접 합성하여 생성

### 전체 결과 요약
- 실제 전화 통화 녹음 → **진짜로 정상 탐지**
- tts 합성 파일 대부분 → **가짜로 정상 탐지**
- 예외 1건(test.mp3) → 가짜인데 진짜로 오분류

### edge-tts 표준 화자 — 정상 탐지

| 파일 | 판별 결과 | 저음역 AI | 음성핵심 AI | 명료도 AI | 고음역 AI |
|------|----------|----------|-----------|---------|---------|
| test2.mp3 | 가짜 **87.89%** | 99.02% | 0.05% | 0.27% | 90.73% |
| test3.mp3 | 가짜 **90.52%** | 98.98% | 0.06% | 0.46% | 83.86% |

공통 패턴: **저음역(0~300Hz)과 고음역(3~8kHz) 모두 90% 이상** → 두 대역에서 동시에 AI 아티팩트 검출되어 정상 탐지

### 미탐지 케이스 — `test.mp3`

| 파일 | 판별 결과 | 저음역 AI | 음성핵심 AI | 명료도 AI | 고음역 AI |
|------|----------|----------|-----------|---------|---------|
| test.mp3 | 진짜 **99.54%** ← 오분류 | 100% | 0.01% | 0.01% | 0.02% |

원인 분석:
- 저음역을 단독으로 들으면 AI 확률 100% → 저음역 자체에는 합성 아티팩트 존재
- 그러나 **음성핵심 / 명료도 / 고음역이 모두 0.01~0.02%** → 중·고음역 합성 품질이 극도로 자연스러움
- GRU가 전 대역을 종합 판단할 때 중·고음역의 "진짜다움"이 저음역 아티팩트를 압도 → 전체 0.46% 가짜로 오분류
- test2/3은 저음역 + 고음역 둘 다 90%+ 이므로 두 대역의 아티팩트가 합쳐져 탐지 가능, test.mp3은 고음역이 사람 수준이라 탈출

### 왜 고품질 TTS는 탐지가 안 되는가

RawNet2는 **ASVspoof2019** 데이터셋으로 사전학습된 모델로, 2019년 이전 TTS 시스템의 아티팩트 패턴을 기준으로 학습됨.

| 구분 | 내용 |
|------|------|
| 모델이 탐지하는 것 | 구형 TTS 보코더의 주파수 아티팩트 (주로 저음역 · 고음역 경계) |
| 최신 Neural TTS의 특성 | 중·고음역(모음·자음) 합성 품질이 실제 사람 수준에 근접 |
| 결과 | 저음역에 아티팩트가 남아있어도, 모델이 "중·고음역이 진짜 같으면 진짜"로 판단 |

→ 학습 데이터에 없던 수준의 고품질 TTS는 모델이 학습한 아티팩트 패턴에서 벗어나 탐지 회피 가능
→ 한국어 파인튜닝으로 일부 보완했으나, edge-tts처럼 자연스러운 최신 모델은 여전히 일부 통과

**고품질 TTS의 특징 (탐지가 어려운 이유)**

| 특성 | 설명 |
|------|------|
| 모음·자음 전환 | 발음 간 전환이 자연스럽고 명료함 |
| 기식음·마찰음 재현 | 고음역의 미세한 숨소리·마찰 소리까지 합성 |
| 프로소디 | 강세·억양·리듬이 자연스러워 딱딱한 느낌 없음 |
| 보코더 아티팩트 없음 | 기계음·울림 같은 합성 흔적이 거의 없음 |

구형 TTS는 위 항목들이 부자연스러워 고음역에서도 아티팩트가 검출됐지만, 최신 Neural TTS는 이를 극복해 중·고음역이 실제 사람 목소리와 구분하기 어려운 수준에 도달.

## Grad-CAM 시행착오

모델이 "왜 가짜인지" 근거를 보여주기 위해 아래 순서로 개선:

| 단계 | 시도 | 문제 | 결과 |
|------|------|------|------|
| 1 | 음향 지표 4종 (librosa) | 모델 내부 근거가 아님 | 보조 지표로만 유지 |
| 2 | Grad-CAM — 주파수 대역별 영향도 | 어느 대역이 중요한지만 알고, AI 특성인지 진짜 특성인지 방향 불명 | 영향도 출력은 유지 |
| 3 | Signed Grad-CAM — 그래디언트 부호로 방향 추출 시도 | 타임스텝마다 부호가 뒤집혀 평균 내면 상쇄 → 불안정 | 폐기 |
| 4 | Ablation — 대역을 직접 마스킹 후 AI 확률 변화(Δ) 측정 | 모델 확신도 높을 때 Δ가 너무 작아 모든 대역이 50% 근처로 몰림 | 정규화 방식 수정 |
| 5 ✅ | Ablation + 상대 정규화 (Δ 최대값 기준 비례 스케일) | — | **채택** |

**부가 버그**: `eval()` 상태에서 GRU 드롭아웃이 그래디언트를 차단 → backward 전후로 `model.gru.train() / eval()` 수동 전환으로 해결

**최종 출력 구조**
- 영향도 (Grad-CAM): 어느 주파수 대역이 판단에 얼마나 관여했는지
- AI 기여도 (Ablation): 해당 대역이 AI 특성인지 진짜 특성인지 방향

## 이슈

### 테스트 데이터 불확실성
- **진짜 파일**: 전화 녹음 파일 사용 → 전화 품질 열화(8kHz 대역 제한, 압축 노이즈) 포함
- **가짜 파일**: 일반 환경 녹음 파일 사용 → 전화 품질 열화 없음
- 두 데이터의 **녹음 환경 자체가 다르기 때문에** 모델이 AI/진짜를 구별하는 게 아니라 전화/비전화를 구별할 가능성 존재

### 처리 계획
- 현재까지의 탐지 결과는 가짜 파일에 전화 환경 열화가 없는 조건에서 나온 것이므로 신뢰도가 불완전
- **다음 테스트**: 가짜 TTS 파일에 전화 채널 시뮬레이션(8kHz 다운샘플 → 16kHz 업샘플)을 적용한 뒤 동일하게 탐지 수행
- 전화 환경을 씌워도 탐지되면 실제 서비스 환경에서도 유효한 모델로 판단 가능
- `korean_dataset.py`의 `_phone_channel` augmentation이 학습 시 이미 적용되어 있으므로 탐지 자체는 유지될 가능성이 높으나 수치 확인 필요
